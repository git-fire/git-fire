## Primary version (recommended)
As a software engineering lead, I kept running into the same problem:
too many local Git repos, too many in-progress changes, and no reliable one-command checkpoint before context switches or risky changes.

So I built **git-fire**: an open-source Go CLI that helps you quickly checkpoint many repos with guardrails.

Repo: https://github.com/git-fire/git-fire

What it focuses on:
- streamed scan -> backup pipeline (starts work before full discovery ends)
- bounded parallel execution for larger repo sets
- persistent repo registry (set once, reuse)
- dry-run planning + one-screen `--status`
- secret detection guardrails before push
- structured JSON logs for audit/automation

Current state is alpha/MVP.
No tagged release yet (first release coming soon), so install is:
`go install github.com/git-fire/git-fire@main`

Latest main also includes a Windows path handling fix that will be part of the first tagged release.

If your build is literally on fire:
1) `git-fire`
2) leave building

Would love feedback from folks running multi-repo workflows (dev, infra, docs, agentic coding sessions).

## Short version (if you want more concise)
Built **git-fire** (OSS Go CLI) to checkpoint many repos in one command when things get chaotic.

It adds guardrails around multi-repo backups: dry-run planning, bounded parallelism, persistent repo registry, secret warnings, and structured logs.

Alpha/MVP today. First tagged release is coming soon.
For now: `go install github.com/git-fire/git-fire@main`
Repo: https://github.com/git-fire/git-fire

## Optional first comment to boost context
Why this exists: manual shell loops across many repos fail in subtle ways (auth/network hiccups, missed repos, inconsistent sequencing). `git-fire` is about repeatability and safer defaults, not replacing clean PR hygiene.
